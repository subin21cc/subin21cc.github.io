본격적인 프로토타입 설계에 앞서, On-Care가 타겟하는 사용자가 누구이고 어떤 흐름으로 앱을 사용하는지를 글로 정리해 두기로 했다. 이 글은 **시나리오 1차 초안(v1)**이며, 앞으로 피그마 프로토타입 → Flutter 구현 → 라이브 데모 배포를 거치며 보강·확정될 예정이다.

오늘 정리한 두 가지는 다음과 같다.

1. **페르소나** — 우리가 풀고자 하는 문제를 정면으로 마주한 단 한 명의 사용자
2. **사용자 액션 시나리오** — 그 사용자가 4개 메인 탭(대시보드 / 식단 / 운동 / MY)에서 어떤 액션을 취하는지

페르소나는 통계가 아니라 **얼굴을 가진 1인**으로 그려야 의사결정 기준이 된다는 디자인 씽킹의 권고를 따랐다. "20~30대 만성질환 위험군"이라는 일반 명사 대신 김민수 씨라는 구체적인 인물을 두면, 기능을 정할 때 "민수 씨가 이걸 진짜 쓸까?"라는 한 줄 질문으로 의사결정이 가능해진다.

# 페르소나

27세 직장인 김민수 씨는 최근 검진에서 당뇨 위험 판정을 받았다. 아침은 편의점, 점심은 구내식당, 저녁은 배달음식이 일상이다. 삼성헬스를 써봤지만 수동 입력이 귀찮아 이틀 만에 포기했고, 어떤 음식이 혈당을 올리는지 알고 싶지만 기존 앱은 칼로리만 알려준다.

# 사용자 액션 시나리오

### 1. [대시보드]

- **건강 상태 모니터링**: 민수 씨는 출근길에 앱을 켜고 대시보드 상단의 **칼로리, 나트륨, 혈당** 프로그레스 바를 확인하여 오늘 남은 영양 섭취 허용량을 체크합니다.
- **일정 확인**: '오늘의 일정' 섹션에서 오후에 예약된 병원 방문 시간을 다시 한번 확인합니다.

### 2. [식단]

- **카메라 호출**: 저녁 식사 전, 화면 중앙의 **[카메라] 버튼**을 눌러 촬영 모드로 진입합니다.
- **식단 촬영**: 배달된 비빔밥 사진을 촬영합니다. 이때 시스템은 음식 이미지만을 인식하여 자동으로 분석을 시작합니다.
- **영양 정보 확인 및 보정**: 분석 결과 창에 뜬 **나트륨 함량**과 영양 성분을 확인합니다. 국의 종류가 잘못 인식되었다면 **[수동 보정]** 버튼을 눌러 수동 조정한 뒤 [저장하기]를 누릅니다.

### 3. [운동]

- **주변 센터 검색**: **[플레이스]** 탭을 눌러 집 근처 헬스장 위치와 리스트를 확인합니다.
- **트레이너 상담 요청**: 마음에 드는 센터를 고른 뒤 **[트레이너 1:1 채팅]** 버튼을 클릭합니다.
- **데이터 공유 및 대화**: 민수 씨의 식단과 운동 로그가 요약된 '건강 데이터 요약본'이 트레이너에게 전송된 것을 확인하고, 별도의 설명 없이 바로 "오늘은 어떤 운동을 하면 좋을까요?"라고 질문하며 상담을 시작합니다.

### 4. [MY]

- **건강 지표 체크**: 하루 일과를 마치고 **[마이]** 탭에 접속하여 체중과 혈당 수치가 기록된 **변화 그래프**를 살펴보며 성취감을 느낍니다.
- **포인트 적립**: 오늘 모든 식단과 운동을 기록하여 지급된 **포인트**를 확인합니다.

## v1의 한계와 다음 단계

이 1차 초안은 의도적으로 **추상 수준이 높다.** 화면의 구체적인 위치, 버튼 라벨, 화면 전환 방식 같은 디테일은 아직 정해지지 않았다. 다음 단계에서 점진적으로 보강한다.

- **피그마 1차 프로토타입** — 4개 탭을 실제 화면으로 옮긴다. 시나리오의 빈틈(예: 카메라 진입 위치, 영양 정보 보정 UI 등)이 시각화 단계에서 드러날 것이다.
- **Flutter MVP 구현** — 시나리오의 액션이 실제 위젯·라우트·API 호출로 매핑된다. 이 단계에서 v1에 없던 보조 화면(AI 코치 / 알림 / 헬스장 트레이너 채팅)이 구체화된다.
- **확정 시연 시나리오(Self-Demo Guide)** — v1을 기준점으로 두고, 실제 빌드된 MVP에서 어떤 흐름이 핵심 차별점을 가장 잘 보여주는지를 5개 시나리오로 재구성한다.

페르소나(김민수 씨)는 위 단계 어디서도 바뀌지 않는다. **시나리오는 진화해도 사용자는 변하지 않는다**는 원칙을 다음 회의에서도 한 번 더 강조해 둘 생각이다.
