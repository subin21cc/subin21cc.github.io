# 섹션 3. 스프링 웹 개발 기초

스프링 부트를 이용한 웹 개발의 세 가지 핵심 방식인 **정적 컨텐츠, MVC와 템플릿 엔진, 그리고 API**에 대해 정리해 보겠습니다. 이 내용은 입문자가 스프링의 동작 원리를 파악하는 데 가장 중요한 기초 체력이 됩니다.

---

## 1. 정적 컨텐츠 (Static Content)

정적 컨텐츠는 서버에서 별도의 처리 없이 파일을 웹 브라우저에 **그대로** 내려주는 방식입니다.

*   **동작 방식**: `/static` 폴더에 있는 HTML 파일을 직접 반환합니다.
*   **특징**: 서버에서 프로그래밍을 할 수 없으며, 단순히 파일을 전달하기만 합니다.
*   **프로세스**:
  1. 웹 브라우저에서 특정 파일(예: `hello-static.html`)을 요청합니다.
  2. 내장 톰캣 서버가 요청을 받고 스프링 컨테이너에 전달합니다.
  3. 스프링은 우선순위를 가진 **컨트롤러**에서 해당 경로가 있는지 찾습니다.
  4. 매핑된 컨트롤러가 없다면 내부의 `resources: static/hello-static.html`을 찾아 반환합니다.

---

## 2. MVC와 템플릿 엔진 (Model, View, Controller)

가장 보편적인 방식으로, 서버에서 프로그래밍을 통해 HTML을 **동적으로 변경**하여 내려주는 방식입니다.

*   **구성 요소**:
  *   **Model**: 데이터와 비즈니스 로직을 담는 객체입니다.
  *   **View**: 화면(UI)을 담당하며, 오직 출력에만 집중합니다.
  *   **Controller**: 사용자 요청을 받고, 모델에 데이터를 담아 뷰로 넘겨주는 역할을 합니다.
*   **동작 방식**:
  1. 컨트롤러에서 리턴 값으로 문자를 반환하면, `viewResolver`가 화면을 찾아서 처리합니다.
  2. 스프링 부트의 템플릿 엔진(Thymeleaf 등)이 HTML을 동적으로 렌더링(변환)한 후 브라우저에 전달합니다.

---

## 3. API (Application Programming Interface)

과거에는 HTML을 직접 내려줬다면, 최근에는 **데이터(JSON)**만 깔끔하게 내려주는 API 방식이 주를 이룹니다. 안드로이드/아이폰 앱과의 통신이나, React/Vue 같은 프론트엔드 프레임워크와 협업할 때 필수적입니다.

*   **핵심 키워드**: `@ResponseBody`
*   **동작 방식**:
  *   메서드에 `@ResponseBody` 어노테이션을 붙이면, HTTP의 BODY에 문자나 객체를 직접 반환합니다.
  *   **문자 반환**: 그냥 문자열이 그대로 전달됩니다.
  *   **객체 반환**: 스프링의 기본 설정에 따라 객체가 **JSON 형식**으로 변환되어 반환됩니다. (기본적으로 `MappingJackson2HttpMessageConverter`가 작동)
*   **장점**: 화면 렌더링을 서버가 고민할 필요 없이, 데이터 전달에만 집중할 수 있습니다.

---

### 💡 요약 및 비교

| 구분 | 정적 컨텐츠 | MVC와 템플릿 엔진 | API |
| :--- | :--- | :--- | :--- |
| **반환 형태** | 파일 그대로 (HTML) | 변환된 HTML | 데이터 (주로 JSON) |
| **서버 연산** | 없음 | 있음 (HTML 생성) | 있음 (데이터 생성) |
| **사용 사례** | 단순 안내 페이지 | 일반적인 웹 사이트 | 앱 개발, SPA(React, Vue) |

> **정리하며**:
> 단순한 페이지라면 **정적 컨텐츠**, 서버에서 동적인 화면을 만들어야 한다면 **MVC**, 화면 없이 데이터만 주고받아야 한다면 **API**를 선택하면 됩니다. 스프링은 `@ResponseBody` 유무에 따라 ViewResolver를 쓸지, HttpMessageConverter를 쓸지 똑똑하게 판단해줍니다.
